Demo Writeup — Simple IDOR Discovery

During testing of a web application, I noticed that user profile data was being loaded using a numeric ID in the URL. After logging in as a normal user, I changed my own ID from /api/profile/102 to /api/profile/103.

The server returned another user’s data without any permission checks. This confirmed an Insecure Direct Object Reference (IDOR) vulnerability.

Impact: An attacker could view information belonging to other users. The fix is to verify ownership on the server side, instead of trusting user-controlled IDs.

Note: This is a demo example only — real reports are always responsibly disclosed.
